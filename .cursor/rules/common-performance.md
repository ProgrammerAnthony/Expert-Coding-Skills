# 性能与上下文

## 模型选择策略

**Haiku 类模型**（能力接近 Sonnet 的约九成，成本显著更低）：

- 轻量、高频调用的子 Agent
- 结对编程与常规代码生成
- 多 Agent 体系中的执行型 Worker

**Sonnet 类模型**（偏主开发）：

- 主力开发与日常编码
- 编排多 Agent 工作流
- 复杂编码任务

**Opus 类模型**（深度推理）：

- 复杂架构决策
- 需要最强推理的任务
- 研究与分析类任务

（具体型号以当前产品为准，按成本与能力权衡选用。）

## 上下文窗口管理

在上下文接近满载（尤其**最后约 20%**）时，避免安排：

- 大规模重构
- 跨多文件的大功能实现
- 复杂交互链路的调试

对上下文占用较温和的任务更适合在窗口紧张时处理：

- 单文件小改动
- 独立工具函数或小模块
- 文档更新
- 简单 Bug 修复

## 扩展思考（Extended Thinking）与计划模式

扩展思考默认可能占用较多 token 用于内部推理。

控制方式示例（以客户端实际能力为准）：

- **开关**：快捷键（如 macOS Option+T / Windows Alt+T）
- **配置**：在 `~/.claude/settings.json` 中设置 `alwaysThinkingEnabled` 等项
- **预算**：如 `export MAX_THINKING_TOKENS=10000`
- **可见性**：部分客户端支持查看思考输出（如 Ctrl+O）

需要深度推理的复杂任务建议：

1. 确认扩展思考已按需开启
2. 启用**计划模式**做结构化拆解
3. 可多轮批判性检查
4. 使用分角色子 Agent 覆盖不同视角

## 构建失败排查

构建报错时：

1. 使用 **build-error-resolver** Agent
2. 逐条分析错误信息
3. 小步修改、逐步验证
4. 每修一处后重新构建确认
