# 常用架构模式与权衡

## 单体架构（Monolith）

**适用场景**：
- 团队 < 10 人
- 业务初期，需求快速变化
- DAU < 10 万，QPS < 1000

**优点**：
- 开发简单，本地调试方便
- 事务处理简单（单数据库）
- 部署简单，无分布式复杂性
- 适合 MVP 阶段

**缺点**：
- 扩展方式粗粒度（整体扩展）
- 代码库膨胀后维护困难
- 部署风险高（一处改动影响全局）

**迁移路径**：模块化单体 → 按业务域抽服务 → 微服务

---

## 模块化单体（Modular Monolith）

**适用场景**：
- 单体已经难以维护，但不想引入分布式复杂性
- 团队 10-30 人，多个小团队协作

**核心原则**：
- 按业务域划分模块（如 users、orders、payments）
- 模块间通过公共接口调用，禁止直接访问其他模块的数据库表
- 每个模块可以独立测试
- 为将来拆分微服务做好准备

```
src/
├── users/              # 用户模块
│   ├── api/           # 对外接口（其他模块通过此调用）
│   ├── service/
│   └── repository/
├── orders/             # 订单模块
│   ├── api/
│   ├── service/
│   └── repository/
└── shared/             # 共享工具（日志、配置、异常）
```

## 微服务架构（Microservices）

**适用场景**：
- 团队 > 50 人，独立团队负责不同服务
- 不同模块有显著不同的扩展需求
- 需要技术多样性（不同服务用不同语言/框架）

**优点**：
- 独立部署，降低部署风险
- 独立扩展，精细化资源利用
- 技术多样性

**缺点（务必充分评估）**：
- 分布式事务复杂（需要 Saga/补偿机制）
- 网络调用引入延迟和可靠性问题
- 服务发现、配置管理、监控成本高
- 本地开发体验差

**⚠️ 陷阱警告**：不要过早微服务化。大多数初期系统从单体开始，在有充分理由时再拆分。

---

## 事件驱动架构（EDA）

**适用场景**：
- 需要服务解耦（发布者不关心消费者）
- 异步处理适合业务（如：发订单后异步发邮件）
- 需要事件溯源（Event Sourcing）

**核心组件**：
- Event Producer（事件发布者）
- Message Broker（Kafka / RabbitMQ / Pulsar）
- Event Consumer（事件消费者）

**注意事项**：
- 消息幂等性（同一消息可能被重复消费）
- 消息顺序保障（只有同一分区内有序）
- 死信队列处理失败消息
- 事件 Schema 版本兼容性

---

## CQRS（命令查询职责分离）

**适用场景**：
- 读写操作的性能要求差异极大
- 读操作需要不同于写操作的数据视图
- 系统读多写少（10:1 以上）

```
写侧（Command）：用户提交命令 → 验证 → 写入主库 → 发布领域事件
                                                    ↓
读侧（Query）：                          消费事件 → 更新读模型 → 查询 API 从读模型返回
```

**与 Event Sourcing 结合**：状态变更存为事件序列，通过重放事件重建状态。

## 架构权衡速查表

| 维度 | 单体 | 微服务 | 选择依据 |
|------|------|--------|----------|
| 开发速度 | ⭐⭐⭐ | ⭐ | 早期选单体 |
| 独立扩展 | ⭐ | ⭐⭐⭐ | 有精细扩展需求选微服务 |
| 部署复杂度 | ⭐⭐⭐ | ⭐ | 运维能力强才选微服务 |
| 分布式事务 | 不需要 | 复杂 | 强一致需求选单体 |
| 技术多样性 | ❌ | ✅ | 有多语言需求选微服务 |
| 团队规模 | 小团队 | 大团队 | 根据团队大小选 |

## 常见架构问题识别

| 症状 | 可能的问题 | 建议方向 |
|------|-----------|----------|
| 查询越来越慢 | 缺少索引、N+1、大数据量 | 缓存、读写分离、分页 |
| 接口超时 | 同步链路过长、外部依赖慢 | 异步化、熔断、超时控制 |
| 一个改动影响全局 | 模块耦合严重 | 模块化、依赖倒置 |
| 代码难以测试 | 副作用混杂、紧耦合 | 依赖注入、接口抽象 |
| 扩容效果不明显 | 存在资源竞争的单点 | 识别瓶颈、水平扩展预设计 |
| 数据不一致 | 缺少事务、并发控制 | 乐观锁、幂等设计 |
