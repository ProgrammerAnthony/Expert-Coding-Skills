# 可删除代码与重构计划模板

## 识别可删除代码

### 立即安全删除的情形

以下类型的代码可在本次 PR 中直接删除：

- **注释掉的代码块**：注释日期超过 1 个月，或无关联 Issue
- **未使用的导入**：通过静态分析确认无引用
- **未使用的函数/方法**：通过全局搜索确认无调用（注意反射调用）
- **冗余的条件分支**：永远为 true/false 的条件
- **重复逻辑**：与其他函数完全相同的实现
- **过时的 TODO**：超过 3 个月无跟进的 TODO 注释

### 延后处理（附计划）的情形

以下代码需要评估影响后再删除：

- **功能开关保护的代码**：确认功能已稳定上线
- **向后兼容的废弃接口**：需要确认所有调用方已迁移
- **A/B 实验代码**：需要确认实验已结束且有结论
- **平台特定的代码路径**：需要在全平台验证后才能删除备用路径

## 重构计划模板

当发现需要重构的代码时，使用以下模板提出计划：

```markdown
### 重构项：[简短描述]

**现状**：
[当前代码的问题描述]

**目标**：
[重构后的期望状态]

**影响范围**：
- 修改文件：[列出文件]
- 影响调用方：[列出调用此代码的地方]
- 需要配套修改：[测试、配置、文档等]

**渐进式步骤**：
1. [第一步：最小改动，可独立上线]
2. [第二步：...]
3. [第三步：...]

**验证方法**：
- [ ] 单元测试覆盖新接口
- [ ] 集成测试验证行为一致
- [ ] 性能基准对比（如有必要）

**风险**：
[潜在风险及缓解措施]
```

## 技术债务优先级矩阵

| 影响面 | 修复成本低 | 修复成本高 |
|--------|-----------|-----------|
| **高影响** | 立即处理（本 PR） | 制定专项计划 |
| **低影响** | 顺手处理 | 记录为低优先级 Issue |

## 常见重构策略参考

| 问题 | 重构手法 |
|------|----------|
| 函数过长 | 提取方法（Extract Method） |
| 参数过多 | 引入参数对象（Introduce Parameter Object） |
| 条件判断复杂 | 以卫语句替换嵌套条件（Replace Nested Conditionals with Guard Clauses） |
| 重复代码 | 提取到工具类或基类 |
| 临时变量过多 | 用查询替换临时变量（Replace Temp with Query） |
| 类型码 | 以策略模式或多态替换（Replace Type Code with Strategy/Polymorphism） |
