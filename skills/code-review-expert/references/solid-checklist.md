# SOLID 原则检查清单

## S — 单一职责原则（SRP）

**核心问题**：这个类/函数是否只做一件事？

检查项：
- [ ] 类是否同时处理业务逻辑和持久化？（应拆分）
- [ ] 函数是否超过 30 行？（考虑拆分）
- [ ] 函数名中是否包含 `and`、`or`（如 `validateAndSave`）？（职责过多）
- [ ] 修改原因是否超过一个？（如"业务变化"或"数据库结构变化"都会触发修改）
- [ ] 测试是否因为需要 mock 太多依赖而变得复杂？

**常见违反模式**：
```python
# 违反：混合验证、业务处理、数据库操作
class UserService:
    def process_user(self, data):
        # 验证
        if not data.get('email'):
            raise ValueError("email required")
        # 业务逻辑
        user = User(**data)
        user.send_welcome_email()
        # 持久化
        db.session.add(user)
        db.session.commit()
```

## O — 开闭原则（OCP）

**核心问题**：添加新功能是否需要修改现有代码？

检查项：
- [ ] 是否通过 `if/elif` 链来区分不同类型的处理？（考虑策略模式）
- [ ] 添加新的支付方式/通知渠道/导出格式是否需要修改核心类？
- [ ] 是否存在 `switch(type)` 或大量 `isinstance` 检查？

**重构方向**：策略模式、工厂模式、插件架构

---

## L — 里式替换原则（LSP）

**核心问题**：子类是否可以安全替换父类？

检查项：
- [ ] 子类方法是否抛出父类不声明的异常？
- [ ] 子类是否缩窄了参数接受范围或扩宽了返回值范围？
- [ ] 子类是否直接 `raise NotImplementedError`？（应考虑接口而非继承）
- [ ] 使用父类的代码，替换为子类后是否行为发生变化？

---

## I — 接口隔离原则（ISP）

**核心问题**：接口是否只包含使用者需要的方法？

检查项：
- [ ] 实现类是否有大量空实现或 `pass` 方法？
- [ ] 客户端是否依赖它从不使用的接口方法？
- [ ] 接口/基类是否超过 5-7 个方法？（考虑拆分）

---

## D — 依赖倒置原则（DIP）

**核心问题**：高层模块是否直接依赖低层实现？

检查项：
- [ ] 业务逻辑类是否直接实例化具体的数据库、HTTP 客户端？
- [ ] 是否在构造函数中 `new` 出依赖而非注入？
- [ ] 单元测试是否因为无法替换依赖而必须用真实数据库/网络？
- [ ] 是否存在 `from concrete_module import ConcreteClass` 在核心业务层？

**重构方向**：依赖注入、接口/抽象类、工厂方法

---

## 架构异味速查

| 异味 | 症状 | 建议 |
|------|------|------|
| 上帝类 | 一个类超过 500 行，知道所有事 | 按职责拆分 |
| 散弹式修改 | 一个变更需要改多个不相关文件 | 提高内聚性 |
| 特性依恋 | 一个方法频繁使用另一个类的数据 | 移动方法 |
| 数据泥团 | 多处出现相同的参数组合 | 封装为对象 |
| 过长参数列 | 函数超过 4 个参数 | 封装为参数对象 |
| 中间人 | 类仅做转发不含逻辑 | 消除或合并 |
