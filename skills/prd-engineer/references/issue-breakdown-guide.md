# GitHub Issues 拆解指南

## Issue 拆解原则

### 粒度标准

**好的 Issue 大小**：
- 开发工作量：2-5 天
- 独立可交付：单独合并后系统仍然正常工作
- 独立可测试：有明确的验收标准
- 边界清晰：团队成员看标题就能理解要做什么

**需要继续拆分的信号**：
- 预估超过 1 周
- 包含多个不相关的功能点
- 无法独立上线（必须和其他 Issue 同时合并）

---

## Issue 标准格式

```markdown
## [类型]: [简洁的功能描述]

**用户故事**：
作为 [用户类型]，我想要 [功能描述]，以便 [业务价值]。

**背景**：
[为什么需要这个 Issue，与整体功能的关系]

**验收标准**：
- [ ] [具体可测试的条件 1]
- [ ] [具体可测试的条件 2]
- [ ] 错误场景：[异常情况的处理]

**技术决策**（开发者决定）：
- [留给开发者的选择空间 1]
- [留给开发者的选择空间 2]

**测试要求**：
- [ ] 单元测试覆盖核心逻辑
- [ ] [特定测试场景]

**超出范围**：
- 本 Issue 不包括 [...]

**估算**：[X 天]
**依赖**：依赖 #[Issue 编号] 完成
**标签**：`feat` `backend` `P0`
```

---

## Issue 类型说明

| 类型标签 | 说明 | 示例 |
|----------|------|------|
| `feat` | 新功能开发 | 实现用户登录 |
| `fix` | 修复缺陷 | 修复密码重置链接失效问题 |
| `refactor` | 重构改善 | 重构用户服务以支持多租户 |
| `test` | 测试补充 | 为支付模块添加集成测试 |
| `docs` | 文档更新 | 更新 API 文档 |
| `perf` | 性能优化 | 优化用户列表查询性能 |
| `chore` | 基础设施 | 升级依赖版本 |

## 从 PRD 到 Issues 的拆解流程

### 1. 识别独立的功能单元

以 PRD 的用户故事为单位，每个 P0/P1 用户故事通常对应 1-3 个 Issue：

```
PRD 用户故事：用户可以通过手机号注册账号

→ Issue #1: feat: 手机号注册接口（后端）
  - 验证手机号格式
  - 发送短信验证码
  - 验证码校验并创建账号

→ Issue #2: feat: 手机号注册页面（前端）
  - 注册表单 UI
  - 倒计时重发逻辑
  - 错误提示展示

→ Issue #3: test: 注册流程集成测试
  - 端到端测试完整注册流程
```

### 2. 识别依赖关系

```
依赖树示例：

Issue #1: 数据库迁移（无依赖，最先）
  ↓
Issue #2: 后端 API（依赖 #1）
Issue #3: 基础组件（无依赖，可并行）
  ↓
Issue #4: 前端页面（依赖 #2 和 #3）
  ↓
Issue #5: 集成测试（依赖 #2 和 #4）
```

### 3. 排优先级

```
P0（必须）：
- 阻塞其他 Issue 的基础设施
- 核心用户故事

P1（重要）：
- 完整功能所需
- 边界条件处理

P2（增强）：
- 体验优化
- 非核心功能
```

## Issues 输出示例

针对"用户注册"功能，拆解如下：

```
## Issue 清单

### P0 优先级（第 1 周）

**#1: feat: 用户数据库模型与迁移**
- 估算：1 天
- 依赖：无

**#2: feat: 发送短信验证码接口**
- 估算：2 天
- 依赖：#1

**#3: feat: 手机号注册接口**
- 估算：2 天
- 依赖：#1, #2

### P1 优先级（第 2 周）

**#4: feat: 注册前端页面**
- 估算：3 天
- 依赖：#3

**#5: test: 注册流程集成测试**
- 估算：2 天
- 依赖：#3, #4

### P2 优先级（第 3 周）

**#6: feat: 注册成功欢迎邮件**
- 估算：1 天
- 依赖：#3
```
