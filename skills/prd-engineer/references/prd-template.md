# PRD 标准模板

使用以下模板输出产品需求文档。按节填写，每节完成后等待用户确认再继续。

---

```markdown
# [功能名称] 产品需求文档

**版本**：v1.0
**作者**：[作者]
**日期**：[YYYY-MM-DD]
**状态**：草稿 / 审查中 / 已批准

---

## 1. 问题陈述

### 1.1 背景
[说明为什么需要这个功能，现有的痛点是什么]

### 1.2 目标用户
[谁会使用这个功能，他们的特征和使用场景]

### 1.3 用户痛点
[用户目前是如何解决这个问题的？现有方案的缺陷是什么？]

---

## 2. 解决方案

### 2.1 核心方案
[高层次描述解决方案，一段话说明做什么]

### 2.2 成功指标
[可量化的成功标准]
- 指标 1：[如：注册转化率提升 X%]
- 指标 2：[如：用户操作步骤从 N 步减少到 M 步]

---

## 3. 功能需求

### 3.1 用户故事

按优先级排序（P0 最高）：

**P0 — 核心功能（必须有）**

**故事 1：[故事标题]**
作为 [用户类型]，
我想要 [功能描述]，
以便 [业务价值]。

*验收标准：*
- [ ] [条件 1]
- [ ] [条件 2]
- [ ] [边界情况：当...时，应该...]
- [ ] [错误场景：如果...失败，应该...]

---

**P1 — 重要功能（应该有）**

**故事 2：[故事标题]**
...（同上格式）

---

**P2 — 锦上添花（可以有）**

**故事 3：[故事标题]**
...（同上格式）

---

### 3.2 非功能性需求

| 类别 | 要求 |
|------|------|
| 性能 | [如：接口响应时间 < 500ms（P99）] |
| 可用性 | [如：功能可用率 > 99.9%] |
| 安全性 | [如：需要登录才能访问，数据需加密] |
| 兼容性 | [如：支持 Chrome/Firefox/Safari 最新两个版本] |
| 国际化 | [如：支持中英文切换] |

---

## 4. 实施决策

*此处列出需要开发团队决定的技术选型，不规定具体实现方式。*

**待决策项：**
- [ ] [决策 1：如选择哪种邮件服务提供商]
- [ ] [决策 2：如数据存储方案]

---

## 5. 测试决策

*此处列出测试策略要求，不规定具体测试代码。*

- 单元测试：[需要哪些核心逻辑覆盖]
- 集成测试：[需要哪些端到端场景]
- 人工验收：[需要 QA 手动验证的场景]

---

## 6. 超出范围（Out of Scope）

**明确不做的事情：**
- [不做 1：如本期不支持批量操作]
- [不做 2：如不支持移动端原生应用]

**推迟到下期的内容：**
- [推迟项 1]

---

## 7. 依赖与风险

### 7.1 依赖
- [依赖项 1：如依赖第三方支付平台接口]

### 7.2 风险
| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| [风险描述] | 高/中/低 | 高/中/低 | [应对方案] |

---

## 8. 参考资料

- [竞品分析链接]
- [用户调研数据]
- [技术文档]
```

---

## 用户故事质量检查

每个用户故事必须满足 INVEST 原则：

| 原则 | 说明 |
|------|------|
| **I**ndependent（独立） | 不依赖其他故事的完成 |
| **N**egotiable（可协商） | 细节可以讨论，不是固定合同 |
| **V**aluable（有价值） | 对用户有明确价值 |
| **E**stimable（可估算） | 开发团队可以估算工作量 |
| **S**mall（小） | 可以在一个迭代内完成 |
| **T**estable（可测试） | 验收标准可以验证 |
