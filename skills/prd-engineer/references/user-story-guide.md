# 用户故事编写指南

## 用户故事的三个层次

### 1. 史诗（Epic）— 大功能方向

过大，无法直接开发。用于战略规划。

```
作为平台卖家，我想要管理我的商品，以便高效运营我的店铺。
```

### 2. 特性（Feature）— 功能模块

可以在 1-2 个迭代内完成。用于 PRD 描述。

```
作为卖家，我想要批量上传商品，以便快速完成店铺初始化。
```

### 3. 故事（Story）— 具体行为

可以在 1-5 天内完成，是 GitHub Issue 的粒度。

```
作为卖家，我想要通过导入 CSV 文件上传最多 100 个商品，
以便一次性完成商品录入而不用逐个手动添加。
```

## 写好用户故事的技巧

### 聚焦用户目标，而非操作步骤

```
差（描述操作）：作为用户，我想要点击"发送"按钮，发送邮件。

好（描述目标）：作为用户，我想要给团队成员发送通知，
以便他们在任务更新时第一时间知晓。
```

### 识别真正的用户类型

不同用户有不同需求，不要用"用户"一概而论：

```
角色区分：
- 作为"新手买家"（第一次购买）：更需要引导和保障
- 作为"资深买家"（多次购买）：更需要效率和便利
- 作为"商家"：更需要批量操作和数据分析
```

### 验收标准的 Given-When-Then 格式

```
场景：[场景描述]
  Given（前提）：[系统或用户的初始状态]
  When（触发）：[用户执行的操作]
  Then（期望）：[系统应该做什么]

示例：
场景：密码过短时注册失败
  Given：用户在注册页面填写信息
  When：用户输入长度为 5 的密码并提交
  Then：系统显示"密码至少需要 8 位"的错误提示
        注册表单保留已填写的信息（除密码外）
        不创建用户账号
```

## 识别和描述边界条件

对于每个功能，系统性地考虑：

| 维度 | 问题 |
|------|------|
| **空值** | 用户不填这个字段会怎样？ |
| **极值** | 用户输入超长内容、超大数字会怎样？ |
| **并发** | 两个用户同时操作同一资源会怎样？ |
| **中断** | 操作进行到一半网络断开会怎样？ |
| **权限** | 不同角色执行此操作会怎样？ |
| **重复** | 重复提交同一操作会怎样？ |

## 实施计划编写（完整模式）

当需要输出实施计划时，提供以下内容：

### 迭代规划建议

```markdown
## 实施计划

### 里程碑 1（约 X 天）：核心功能
- [ ] Issue #1: [标题]（P0，预估 X 天）
- [ ] Issue #2: [标题]（P0，预估 X 天）

### 里程碑 2（约 X 天）：增强功能
- [ ] Issue #3: [标题]（P1，预估 X 天）

### 依赖关系
- Issue #3 依赖 Issue #1 完成
- Issue #4 可以和 Issue #2 并行

### 关键决策点
在开始 Issue #X 前，团队需要讨论并决定：
- [决策 1]
- [决策 2]
```

### 风险与缓解

```markdown
### 风险识别
| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 第三方 API 不稳定 | 功能不可用 | 实现降级方案 |
| 数据迁移量大 | 上线延迟 | 提前一周开始迁移演练 |
```
