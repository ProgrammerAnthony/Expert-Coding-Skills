# 技能架构设计原则

## 文件职责分离

```
SKILL.md          # 导演：告诉 AI 做什么，何时加载什么
references/       # 演员：提供具体的领域知识和检查清单
assets/           # 道具：用于输出的文件，不影响推理
```

### SKILL.md 应该包含

- 工作流（步骤清单）
- 铁律（最重要的约束）
- 确认门控（何时等待用户）
- 加载指令（何时加载哪个 reference）
- 输出格式（最终输出的结构）
- 反模式清单（不该做什么）

### SKILL.md 不应该包含

- 完整的检查清单（放 references）
- 详细的规范说明（放 references）
- 代码示例（放 references，除非非常关键）
- 背景知识介绍（Claude 已经知道的）

---

## 500 行铁律

SKILL.md 超过 500 行通常意味着：
- 包含了应该在 references 中的重型内容
- 包含了 Claude 已经知道的废话
- 工作流设计过于复杂（考虑拆分为多个技能）

**重构方法**：
1. 找出最长的章节
2. 判断它是"导演（流程）"还是"演员（知识）"
3. "演员"内容提取到 `references/` 下的独立文件
4. 在 SKILL.md 中用一行替代："加载 `references/xxx.md`"

---

## References 组织原则

### 好的 Reference 文件

- **聚焦**：一个文件只讲一个主题
- **实用**：以表格、清单、代码示例为主，减少解释性文字
- **独立**：可以单独读懂，不依赖其他 reference
- **命名清晰**：`security-checklist.md` 比 `ref1.md` 好

### Reference 文件大小

| 文件大小 | 评估 |
|---------|------|
| < 100 行 | 考虑合并到相关文件 |
| 100-300 行 | 理想大小 |
| 300-500 行 | 可接受，检查是否需要拆分 |
| > 500 行 | 应该拆分为多个文件 |

---

## 技能复杂度控制

### 何时拆分技能

一个技能变得过于复杂时的信号：
- 有超过 5-6 个主要步骤
- 不同场景的流程差异很大（>50%）
- 用户需要记住超过 3 个命令才能使用

**拆分方式**：
- 按场景拆分（如：全新系统 vs 现有系统优化）
- 按深度拆分（快速版 vs 深度版）
- 按角色拆分（前端 vs 后端代码审查）

## 按需加载策略

**加载时机示例**：

```markdown
## 工作流

### Step 1：收集信息
[不加载任何 reference，直接开始]

### Step 2：架构分析
如果涉及安全问题：加载 `references/security-checklist.md`
如果涉及性能问题：加载 `references/performance-checklist.md`

### Step 3：输出报告
加载 `references/report-template.md`
```

**原则**：
- 总是用到的内容 → 可以在 SKILL.md 里直接写（如果简短）
- 特定步骤才用的内容 → 在该步骤加载
- 大型检查清单 → 始终放 references，按需加载

---

## 最小可行技能（MVP Skill）

初次创建技能时，先做最简单的版本：

```
MVP Skill 结构：
├── SKILL.md（100-200 行）
│   ├── description
│   ├── 铁律（1-2 条）
│   └── 核心工作流（3-5 步）
└── references/
    └── main-checklist.md（如果有复杂检查清单）
```

然后根据实际使用情况迭代：
- 用户经常问的问题 → 加入工作流
- 经常需要参考的规范 → 加入 references
- 经常需要重申的约束 → 加入铁律

---

## 技能与技能之间的关系

相关技能可以相互引用，但要避免循环依赖：

```markdown
# 在 SKILL.md 中引用另一个技能

完成代码审查后，如果发现安全问题，建议使用 /安全审计 进行深度扫描。
```

**原则**：引用只能是"推荐"，不能是"强制"（用户决定是否跳转到另一个技能）。
