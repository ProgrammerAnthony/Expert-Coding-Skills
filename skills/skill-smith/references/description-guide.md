# Frontmatter Description 编写指南

## 为什么 Description 很重要

AI Agent 的技能系统通过 `description` 字段来决定何时自动触发技能。写得好的 description 意味着：
- 用户不需要记住技能名字，自然语言就能触发
- 覆盖更多实际使用场景
- 减少用户的认知负担

---

## 描述结构模板

```yaml
description: "
[核心价值，一句话，说明解决什么问题，产出什么结果]。
触发词：[关键词 1]、[关键词 2]、[关键词 3]、[英文词]、[同义词]、[使用场景描述]...
"
```

**示例（好）**：

```yaml
description: "对当前 git 变更进行资深工程师视角的结构化代码审查，覆盖 SOLID 原则、安全漏洞、性能问题和可删除代码。触发词：代码审查、code review、审查代码、检查代码、代码质量、代码评审、review代码、帮我审查、看看这段代码。"
```

**示例（差）**：

```yaml
description: "Code review skill for reviewing code."
# 问题：太简短，触发词不足，无法在多种场景下被识别
```

## 关键词轰炸技术（Keyword Bombing）

目标：覆盖用户可能说的所有表达方式。

### 中英文双覆盖

```
代码审查 / code review
安全审计 / security audit  
测试驱动 / test driven development / TDD
需求文档 / product requirements document / PRD
```

### 动词变体

```
创建 / 新建 / 编写 / 制作 / 开发 / 设计 / 起草 / 生成
分析 / 审查 / 检查 / 评审 / 扫描 / 诊断
优化 / 改进 / 重构 / 修复 / 完善
```

### 场景描述（不只是名词）

```
"帮我写" / "帮我检查" / "如何设计" / "怎么实现"
"有问题" / "报错了" / "不工作" / "出 bug 了"
```

---

## 触发词数量建议

| 技能类型 | 触发词数量 | 原因 |
|----------|-----------|------|
| 通用工具型 | 15-25 个 | 用法场景多样 |
| 特定流程型 | 10-15 个 | 使用场景相对固定 |
| 元技能（创建技能） | 15-20 个 | 需要覆盖创建/设计/改进等多种意图 |

---

## 描述中不需要的内容

- ❌ 技能的内部实现细节（用户不关心）
- ❌ 完整的功能列表（太长，影响触发效果）
- ❌ 前提条件或安装说明（放 README）
- ❌ 技术术语（普通用户可能不会使用）

---

## 测试你的 Description

完成后做以下测试：
1. 想象 5 种不同的用户说法，看是否都包含在触发词中
2. 检查是否有中文同义词（如"审查"和"检查"）
3. 确认包含了英文关键词（英文项目中也能触发）
4. 确认包含了一些"描述症状"的词（如"报错"→触发调试技能）
