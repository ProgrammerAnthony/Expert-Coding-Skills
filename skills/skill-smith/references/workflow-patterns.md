# 工作流设计模式

## 勾选清单模式（推荐）

将工作流表达为可追踪的清单，让 AI 和用户都能看到进度：

```markdown
## 工作流进度

- [x] Step 1：理解需求（已完成）
- [ ] Step 2：方案设计（进行中）
  - [ ] 2.1 生成选项
  - [ ] 2.2 用户确认
- [ ] Step 3：实施
- [ ] Step 4：验证
```

**好处**：
- AI 不会跳过步骤
- 用户清楚地知道当前在哪个阶段
- 多轮对话中容易恢复状态

---

## 确认门控（Confirmation Gate）

在关键决策点，必须等待用户明确批准才继续。

```markdown
### 设计方案

[展示设计内容...]

---
**在用户确认以上设计前，不进行任何实施操作。**

请选择：
1. 批准方案，开始实施
2. 修改 [某部分]
3. 重新设计
```

**何时使用确认门控**：
- 将要修改文件之前
- 呈现多个方案让用户选择时
- 完成规划阶段，进入实施阶段时
- 准备输出最终结果时

---

## 渐进式加载（Progressive Loading）

不要在技能开始时一次性加载所有 references：

```markdown
# 技能名称

## 工作流

### Step 1：分析
[直接开始，不加载任何 references]

### Step 2：检查安全
加载 `references/security-checklist.md`
[在需要时才加载安全清单]

### Step 3：输出报告
加载 `references/report-template.md`
[在输出时才加载模板]
```

**好处**：节省上下文窗口，只在需要时消耗 token。

---

## 模式选择模式

让用户在开始时选择模式，后续流程根据模式不同：

```markdown
启动时询问：

```
请选择工作模式：
1. 快速模式（5分钟）— 基础检查
2. 标准模式（30分钟）— 全面分析
3. 深度模式（2小时）— 最详细
```

根据选择调整后续步骤的深度和范围。
```

---

## 收集-处理-输出模式

最通用的技能结构：

```markdown
### 收集阶段（获取足够信息）
- 通过提问从用户处收集
- 通过工具调用（git diff、读文件、搜索）收集

### 处理阶段（分析和决策）
- 内部分析，不输出中间状态
- 加载相关 references

### 输出阶段（结构化输出）
- 使用固定模板输出
- 加载 report-template.md
```

---

## 迭代精化模式

对于需要多轮打磨的输出：

```markdown
### 初稿
[输出初始版本]

---
**以上为初稿，请提供反馈：**
1. 需要修改 [某部分]
2. 在 [某处] 添加更多细节
3. 满意，可以完成

[根据用户反馈迭代，直到用户满意]
```

---

## 工作流设计检查清单

设计好工作流后检查：

- [ ] 每一步的输入和输出是否清晰？
- [ ] 是否有不必要的步骤？（删除它）
- [ ] 关键决策点是否有确认门控？
- [ ] 用户何时参与，何时 AI 自主执行？是否合理？
- [ ] references 是否在正确的步骤按需加载？
- [ ] 异常情况（用户输入不足、代码库为空等）是否有处理？
